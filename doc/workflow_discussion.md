# 数据先行 vs 设计先行 — 流程讨论

## 你的想法 vs 现在的设计

```
现在的设计（设计先行）:
  用户说需求 → AI 用假数据生成看板 → 用户连接真实数据 → 导出

你的想法（数据先行）:
  用户先配置数据源 → AI 看到真实数据 → AI 挑选合适图表组成看板 → 导出
```

**我的判断：你的想法更好。** 原因：

| | 设计先行（现在） | 数据先行（你的想法） |
|--|-----------------|-------------------|
| AI 生成依据 | 猜测（"电商大概需要折线图吧"） | 真实数据结构（"orders 表有 date 和 amount，适合折线图"） |
| 生成后的工作量 | 大（要逐个组件连数据、改映射） | 小（已经连好了） |
| 出错概率 | 高（AI 猜的图表类型可能和数据不匹配） | 低（基于真实 schema 选择） |
| 用户体验 | 先看到假数据，再替换 | 一开始就看到真数据 |
| 智能程度 | 低（模板化） | 高（数据驱动决策） |

## 你的流程详细走一遍（帮你理清楚）

```
Step 0: 配置数据源（新增步骤，在设计之前）
  ┌─────────────────────────────────────────────┐
  │ 数据源配置页面                                │
  │                                             │
  │  [+ 添加 API]  [+ 添加数据库]                 │
  │                                             │
  │  已添加:                                     │
  │  ✅ GET /api/v1/sales    → 5个字段           │
  │  ✅ GET /api/v1/orders   → 8个字段           │
  │  ✅ MySQL mydb.orders    → 12个字段          │
  │  ✅ MySQL mydb.products  → 6个字段           │
  │                                             │
  │  AI 已了解你的数据结构，可以开始设计看板        │
  │                    [开始设计 →]               │
  └─────────────────────────────────────────────┘

  添加 API 时:
    用户输入 URL → 后端调用一次 → 拿到返回结构
    → AI 知道: "这个 API 返回 {date, amount, category, region}"

  添加数据库时:
    用户输入连接信息 → 后端连接 → 列出所有表和字段
    → AI 知道: "orders 表有 id, amount, created_at, status, ..."

Step 1: 用户说需求，AI 基于真实数据生成
  用户: "帮我做一个销售数据看板"
  
  AI 场景 Agent 收到的上下文（和现在不同！）:
  {
    "message": "做一个销售数据看板",
    "dataSources": [
      { "name": "sales", "type": "api", "url": "/api/v1/sales",
        "fields": ["date", "amount", "category", "region"] },
      { "name": "orders", "type": "database", "table": "orders",
        "fields": ["id", "amount", "created_at", "status", "customer_name"] }
    ]
  }

  AI 的决策过程:
  → 看到 date + amount → 适合折线图（趋势）
  → 看到 category → 适合饼图（占比）
  → 看到 region → 适合排行榜
  → amount 是数值 → 适合 KPI 卡（汇总）
  → 生成看板，每个组件已绑定真实数据源和字段映射

Step 2: 看板立即显示真实数据
  → 折线图显示真实的销售趋势
  → KPI 卡显示真实的总金额
  → 用户看到的就是最终效果

Step 3: 用户微调
  → "把饼图改成柱状图"
  → "加一个按地区的排行榜"
  → 手动拖拽调整位置
  → 所有调整都基于真实数据

Step 4: 保存或导出
  保存 → 下次打开直接显示最新数据（实时看板）
  导出 → 独立项目，部署后自动从数据源拉取数据
```

## 和现在设计的冲突点

### 冲突 1: UI 流程变了

```
现在: 打开 → 直接进工作区 → 开始设计
改后: 打开 → 数据源配置（或选已有项目）→ 进工作区
```

**解决方案**: 加一个"项目首页"或"起步页"

```
┌────────────────────────────────────────┐
│  AI 看板平台                           │
│                                        │
│  [新建项目]         [打开已有项目]       │
│                                        │
│  新建项目流程:                          │
│  1. 输入项目名                         │
│  2. 配置数据源（可跳过，用 mock）       │
│  3. 进入工作区                         │
│                                        │
│  已有项目:                             │
│  📊 电商销售看板    2026-04-23         │
│  📊 运营日报       2026-04-20         │
└────────────────────────────────────────┘
```

注意：数据源配置"可跳过"——不强制。跳过就用 mock 数据（设计先行模式）。

### 冲突 2: 场景 Agent 的 Prompt 变了

```
现在: AI 只知道用户说了什么 → 根据常识猜测组件
改后: AI 知道用户说了什么 + 有哪些数据 → 基于数据选择组件
```

**影响**: scene.md prompt 模板需要重新设计

```markdown
# 现在的 prompt（简化）
你是看板设计专家。用户需要：{{message}}
请生成组件列表，包含 type, props, position, size。

# 改后的 prompt（简化）
你是看板设计专家。
用户需要：{{message}}
用户已有以下数据源：
{{#each dataSources}}
- {{name}}({{type}}): 字段 {{fields}}，示例数据 {{sample}}
{{/each}}
请根据数据源选择最合适的图表，每个组件必须绑定一个数据源。
```

这不是冲突，是**增强**。Prompt 更丰富，AI 输出更精准。

### 冲突 3: 需要"项目"概念

```
现在: 没有项目管理，就是一个画布
改后: 需要保存/加载项目（包括数据源配置+组件配置）
```

**影响**: 需要加 stores/project.js 管理项目元数据

### 冲突 4: 导出逻辑更复杂（但更有价值）

```
现在: 导出时从 widget props 提取 mock 数据
改后: 导出时根据真实数据源生成后端代码
  - API 源 → 生成代理路由
  - DB 源 → 生成查询路由 + 连接配置
```

## 两种模式并存

建议支持两种起步方式:

```
模式 A: 数据先行（推荐，你的想法）
  配置数据源 → AI 基于真实数据设计 → 所见即所得 → 导出即可用

模式 B: 设计先行（保留，用于原型/演示）
  直接进入工作区 → AI 用 mock 数据设计 → 后续可连接数据 → 导出

用户在新建项目时选择:
  [从数据开始] — 我已有 API 或数据库
  [从模板开始] — 我想先设计再连数据
```

## 关于"问数"功能

你说的"问数"非常有价值，且和现在的架构很契合:

```
用户: "上个月哪个地区销售额最高？"
  ↓
AI 数据 Agent:
  1. 分析问题 → 需要按地区汇总 + 时间过滤 + 排序
  2. 生成 SQL: SELECT region, SUM(amount) FROM orders 
     WHERE created_at >= '2026-03-01' GROUP BY region ORDER BY 2 DESC
  3. 执行查询 → 得到结果
  4. 选择图表: 柱状图(对比) 或 排行榜
  5. 在画布上创建组件展示答案

和现在架构的关系:
  → 就是 "组件 Agent(create)" + "数据 Agent" 的组合
  → 不需要新架构，只需要一个新的复合流程
  → 放在 Phase 2 做，完全合理
```

## 总结

| 问题 | 回答 |
|------|------|
| 你的想法对不对？ | **对，而且比现在的设计更好** |
